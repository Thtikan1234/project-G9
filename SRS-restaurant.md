# Software Requirements Specification 

---
## Restaurant Management System
---

## ระบบจัดการร้านอาหารเบื้องต้น
---

## 1. บทนำ 
---

### 1.1 วัตถุประสงค์ 
เอกสารนี้อธิบายข้อกำหนดความต้องการซอฟต์แวร์สำหรับระบบจัดการร้านอาหาร โดยมีวัตถุประสงค์เพื่อ:

- จัดการการสั่งอาหารและเครื่องดื่ม
- ดูแลระบบการชำระเงิน
- สร้างรายงานและวิเคราะห์ข้อมูล

### 1.2 ผู้มีส่วนได้ส่วนเสีย 
- เจ้าของร้าน / ผู้จัดการร้าน  
- พนักงานหน้าร้าน 
- พนักงานครัว 
- ลูกค้า  
- ทีมพัฒนา / Tester  

### 1.3 ขอบเขตของระบบ 
ระบบจะครอบคลุมการทำงานหลักดังนี้:

- การจัดการเมนูอาหาร
- การรับออเดอร์
- การชำระเงิน
- การจัดการโต๊ะอาหาร


## 2. ความต้องการด้านฟังก์ชันการทำงาน


### 2.1 การจัดการเมนูอาหาร

#### 2.1.1 การจัดการรายการอาหาร
1. การเพิ่มเมนูอาหาร:

-  ชื่อเมนู (ไทย/อังกฤษ)
-  หมวดหมู่อาหาร
-  ราคา
-  รูปภาพ
-  เมนูแนะนำ/โปรโมชัน
-  สถานะการขาย

2. การแก้ไขเมนู:

- ปรับปรุงข้อมูลเมนู
- เปลี่ยนราคา
- อัพเดทรูปภาพ
- ปรับสถานะการขาย

3. การจัดหมวดหมู่:
- สร้างหมวดหมู่อาหาร
-  จัดเรียงลำดับการแสดงผล


### 2.2 คุณลักษณะผู้ใช้ 
| ผู้ใช้ | คุณลักษณะ | สิทธิ์ |
|-------|-------------|---------|
| ลูกค้า | ใช้งานง่าย ไม่จำเป็นต้องมีความรู้ด้านคอมพิวเตอร์ | ดูเมนู, สั่งอาหาร |
| พนักงานหน้าร้าน | ใช้ระบบพื้นฐานเป็น | รับออเดอร์, ชำระเงิน |
| พนักงานครัว | ต้องดูรายการอาหารแบบเรียลไทม์ | รับออเดอร์, อัปเดตสถานะ |
| ผู้ดูแลระบบ | รู้จักการตั้งค่าระบบ | จัดการเมนู, ดูรายงาน |



### 2.3 การชำระเงิน

#### 2.3.1 การคิดเงิน
1. การออกบิล:

-  สรุปรายการอาหาร
-  คำนวณภาษี
-  ส่วนลด
-  แยก/รวมบิล

2. การรับชำระเงิน:

-  เงินสด
-  บัตรเครดิต/เดบิต
-  Mobile Banking
-  E-Wallet
-  คูปอง/บัตรกำนัล

#### 2.4.2 การจัดการเอกสาร
1. การออกเอกสาร:
-  ใบเสร็จรับเงิน
-  ใบกำกับภาษี
-  ใบสรุปยอดประจำวัน
-  รายงานการขาย


### 2.5 Dashboard
#### 2.5.1 Dashboard ภาพรวม
1. สรุปการขาย:

-  ยอดขายประจำวัน
-  เมนูขายดี

2. สถานะการทำงาน:

-  สถานะโต๊ะ
-  คิวอาหาร


### 2.6 การออกรายงาน
1. รายงานการขาย:

-  รายงานประจำเดือน
-  รายงานตามประเภทอาหาร
-  รายงานตามพนักงาน

2. รายงานการเงิน:

-  รายงานภาษี
-  รายงานต้นทุน
-  รายงานกำไร-ขาดทุน
-  รายงานการชำระเงิน

3. รายงานสต็อก:

-  รายงานสินค้าคงเหลือ
-  รายงานการสั่งซื้อ

4. สมมติฐานและข้อจำกัด 
- ระบบทำงานบนเว็บเบราว์เซอร์  
- ต้องมีอินเทอร์เน็ตหรือ LAN ภายในร้าน  
- อุปกรณ์ POS/Tablet มีหน้าจอที่รองรับเว็บแอป  


## 3. ความต้องการที่ไม่ใช่ฟังก์ชันการทำงาน 

### 3.1 ประสิทธิภาพ
1. รองรับผู้ใช้งานพร้อมกันได้อย่างน้อย 20 คน
2. เวลาตอบสนองไม่เกิน 2 วินาที
3. ทำงานได้ 24 ชั่วโมง

### 3.2 ความปลอดภัย
1. ระบบยืนยันตัวตนผู้ใช้งาน
2. เข้ารหัสข้อมูลสำคัญ
3. บันทึกประวัติการใช้งาน
4. สำรองข้อมูลอัตโนมัติ

### 3.3 ความสามารถในการใช้งาน
1. ใช้งานง่าย เป็นภาษาไทย
2. รองรับการทำงานบนแท็บเล็ต
3. มีระบบช่วยเหลือ
4. แสดงผลได้ทุกขนาดหน้าจอ

### 3.4 User Interface
- หน้าเมนูอาหาร  
- หน้าสร้างออเดอร์  
- หน้าแสดงสถานะออเดอร์ครัว  
- หน้าแคชเชียร์สำหรับชำระเงิน  
- หน้า Admin Dashboard

### 3.5 Hardware Interface
- เครื่องพิมพ์ใบเสร็จ  
- Tablet / PC หน้าร้าน  
- หน้าจอครัว 

### 3.6 Software Interface
- ระบบชำระเงิน (ถ้ามี)  
- ฐานข้อมูล MySQL  


## 4. ข้อจำกัดของระบบ
1. ต้องใช้อินเทอร์เน็ตสำหรับบางฟังก์ชัน
2. รองรับเฉพาะ iOS และ Android รุ่นใหม่
3. เก็บข้อมูลย้อนหลัง 5 ปี
4. ปฏิบัติตาม พ.ร.บ. คุ้มครองข้อมูลส่วนบุคคล

## 5. Functional Requirements

### 5.1 การจัดการเมนู (Menu Management)

| รหัส | รายละเอียด |
|----|-------------|
| FR-1 | ผู้ดูแลระบบสามารถเพิ่มเมนูใหม่ได้ |
| FR-2 | ผู้ดูแลระบบสามารถแก้ไขชื่อ ราคา และรายละเอียดเมนูได้ |
| FR-3 | ผู้ดูแลระบบสามารถลบเมนูได้ |
| FR-4 | ระบบต้องแสดงเมนูทั้งหมดให้ลูกค้าดูได้ |


### 5.2 ระบบสั่งอาหาร (Order Processing)

| รหัส | รายละเอียด |
|----|-------------|
| FR-5 | ลูกค้าหรือพนักงานสามารถเลือกเมนูและเพิ่มลงในออเดอร์ได้ |
| FR-6 | ระบบต้องสร้างหมายเลขออเดอร์โดยอัตโนมัติ |
| FR-7 | ระบบต้องส่งออเดอร์ไปยังครัวทันทีหลังจากยืนยันคำสั่งซื้อ |


### 5.3 ระบบจัดการครัว (Kitchen Management)

| รหัส | รายละเอียด |
|----|-------------|
| FR-8 | พนักงานครัวต้องสามารถเห็นรายการออเดอร์ทั้งหมด |
| FR-9 | พนักงานครัวสามารถอัปเดตสถานะอาหาร เช่น กำลังทำ / เสร็จแล้ว |


### 5.4 ระบบการชำระเงิน (Payment System)

| รหัส | รายละเอียด |
|----|-------------|
| FR-10 | ระบบสามารถคำนวณยอดรวมทั้งหมดของออเดอร์ได้ |
| FR-11 | ระบบสามารถพิมพ์หรือแสดงใบเสร็จได้ |
| FR-12 | ระบบรองรับการชำระเงินหลายประเภท เช่น เงินสด และ QR Code |


### 5.5 รายงาน (Reporting)

| รหัส | รายละเอียด |
|----|-------------|
| FR-13 | ผู้ดูแลระบบสามารถดูรายงานยอดขายรายวัน รายสัปดาห์ และรายเดือน |
| FR-14 | ระบบสามารถส่งออกข้อมูลยอดขายเป็นไฟล์ CSV หรือ PDF ได้ |


## 6. ข้อกำหนดที่ไม่เกี่ยวข้องกับการทำงาน

### 6.1 Performance
- ระบบโหลดหน้าเมนูภายใน 3 วินาที  
- รองรับผู้ใช้งานพร้อมกันอย่างน้อย 20 คน  

### 6.2 Security
- ผู้ใช้ต้องล็อกอินก่อนใช้งานระบบภายใน  
- ข้อมูลผู้ใช้และออเดอร์ต้องถูกจัดเก็บอย่างปลอดภัยในฐานข้อมูล  

### 6.3 การใช้งาน
- UI ต้องใช้งานง่าย เหมาะกับผู้ใช้ไม่มีพื้นฐาน  
- รองรับหน้าจอ Tablet  

### 6.4 ความน่าเชื่อถือ
- ระบบต้องพร้อมใช้งาน (Uptime ≥ 95% ในเวลาทำการ)  


## 7. เกณฑ์การยอมรับ

- สามารถสั่งอาหารได้สำเร็จและออเดอร์ไปถึงครัว  
- ครัวสามารถอัปเดตสถานะอาหารและลูกค้าเห็นสถานะได้  
- ระบบสร้างใบเสร็จถูกต้อง  
- ผู้ดูแลระบบสามารถจัดการเมนูและดูรายงานยอดขายได้  


## 8. Glossary (พจนานุกรมคำศัพท์)
- **Order** – การสั่งอาหาร  
- **Menu Item** – รายการอาหารหรือเครื่องดื่ม  
- **Bill** – ใบเสร็จ/ยอดชำระ  
- **Admin** – ผู้ดูแลระบบที่จัดการข้อมูลในระบบ
